# VALOR Desktop Client – Build & Release

## Overview

Every production build goes through `scripts/build-win.mjs`, which:
1. Bumps the version in `package.json`
2. Generates the NSIS installer BMP assets
3. Compiles the Electron app with `electron-vite`
4. Packages a signed NSIS installer with `electron-builder`
5. (Optional) Uploads the installer + `latest.yml` to the update server

---

## npm scripts

| Command | What it does |
|---------|-------------|
| `npm run dev` | Start in development mode with hot reload |
| `npm run build` | Compile only (no packaging) |
| `npm run build:win` | Full build — patch version bump → installer |
| `npm run build:win:minor` | Full build — minor version bump |
| `npm run build:win:major` | Full build — major version bump |
| `npm run build:win:nobump` | Full build — keep current version (re-package) |
| `npm run release:win` | Patch bump → build → **upload to update server** |
| `npm run package:win` | Quick build without the pipeline script |
| `npm run package:linux` | Build Linux AppImage + .deb |

---

## Version numbering

VALOR follows **semantic versioning** (`MAJOR.MINOR.PATCH`):

| Bump | When to use |
|------|-------------|
| `patch` | Bug fixes, small tweaks. Use for most releases. |
| `minor` | New features that are backwards-compatible. |
| `major` | Breaking changes or significant redesigns. |

The build script updates `package.json` automatically. The running app reads its version from `package.json` at bundle time; `electron-updater` compares it against the hosted `latest.yml` to decide if an update is available.

## Build pipeline — step by step

```
npm run build:win
        │
        ├─ [1/4] npm version patch --no-git-tag-version
        │         Updates version in package.json  (e.g. 0.1.0 → 0.1.1)
        │
        ├─ [2/4] node scripts/generate-assets.mjs
        │         Writes build/installerSidebar.bmp  (164×314 dark BMP)
        │         Writes build/installerHeader.bmp   (150×57  dark BMP)
        │
        ├─ [3/4] npm run build  (electron-vite build)
        │         Compiles main process    →  out/main/index.js
        │         Compiles preload script  →  out/preload/index.js
        │         Bundles React renderer   →  out/renderer/
        │
        └─ [4/4] electron-builder --win --publish never
                  Packages NSIS installer  →  dist/VALOR-Setup-0.1.1.exe
                  Generates update manifest → dist/latest.yml  (not uploaded yet)
```

## Release pipeline (publishing updates)

```
npm run release:win
        │
        ├─ [1–4] same as build:win above
        │
        └─ [5/5] electron-builder --win --publish always
                  Uploads VALOR-Setup-0.1.1.exe  to https://valor.dawn-star.co.uk/updates/
                  Uploads latest.yml             to https://valor.dawn-star.co.uk/updates/
```

### What the update server needs

The update server is just a static file host at `https://valor.dawn-star.co.uk/updates/`. It must serve two files:

| File | Description |
|------|-------------|
| `latest.yml` | Metadata: version, SHA-512 hash, file size, download URL. Generated by electron-builder. |
| `VALOR-Setup-<version>.exe` | The installer itself. |

**Minimal Express setup on the VALOR backend:**
```javascript
// In backend/src/index.js — serve update files as static assets
app.use('/updates', express.static(path.join(__dirname, '../releases')))
```

Then copy `dist/latest.yml` and `dist/VALOR-Setup-<version>.exe` into `backend/releases/` on the server after each release.

### Auto-upload with `--publish always`

For `--publish always` to push files automatically, you need a credential configured. With the `generic` provider, electron-builder doesn't auto-upload — you must copy files manually. The `--publish always` flag is included in the script for future use if you switch to GitHub Releases or an S3 provider.

**For now, the release workflow is:**
1. `npm run build:win` (local build)
2. Manually copy `dist/latest.yml` and `dist/VALOR-Setup-<version>.exe` to the server's `releases/` folder.

## NSIS installer details

The installer wizard has these pages (in order):

1. **Welcome** — Custom heading + body text from `build/installer.nsh`
2. **License** — RTF license from `build/license.rtf`
3. **Install location** — Default `%LocalAppData%\Programs\VALOR`, user-changeable
4. **Installing** — Progress bar while files are extracted
5. **Finish** — "Launch VALOR now" checkbox + link to valor.dawn-star.co.uk

### Branding assets

| File | Size | How generated |
|------|------|--------------|
| `build/installerSidebar.bmp` | 164×314 px | `scripts/generate-assets.mjs` — dark BMP with red stripe |
| `build/installerHeader.bmp` | 150×57 px | `scripts/generate-assets.mjs` — dark BMP with red border |
| `build/installer.nsh` | — | Hand-written NSIS includes (welcome/finish text, links) |
| `build/license.rtf` | — | RTF license document |
| `build/icon.ico` | 256×256 | **You must supply this** — not in repo |

To replace the BMP images with custom designs, either:
- Edit the pixel functions in `scripts/generate-assets.mjs` (pure Node.js drawing).
- Replace the generated files with hand-crafted BMPs from a design tool (must be 24-bit uncompressed BMP, exact pixel dimensions).

## Code signing (Windows)

Unsigned Windows installers trigger a **SmartScreen** warning on first run ("Windows protected your PC"). The app will still install and run, but users must click "More info → Run anyway".

To eliminate this warning, sign the installer with an EV code-signing certificate:

1. Obtain a certificate (DigiCert, Sectigo, etc.) — costs ~£300–£500/year.
2. Configure in `electron-builder.yml`:
   ```yaml
   win:
     certificateFile: path/to/cert.pfx
     certificatePassword: ${env.CERT_PASSWORD}
   ```
3. Run the build — electron-builder signs the installer automatically.

Code signing is not required for internal/trusted users but is recommended for wider distribution.

## Known build issues & solutions

This section documents real issues encountered during development and how they were resolved.

### 1. `electron-vite: not recognized as a command`

**Symptom:** Running `npm run dev` or `npm run build` fails immediately with:
```
'electron-vite' is not recognized as an internal or external command
```

**Cause:** `node_modules/.bin` is missing — `npm install` was never run (or the folder was deleted).

**Fix:** The build pipeline (`scripts/build-win.mjs`) auto-detects this and runs `npm install` automatically before continuing. If you hit this in dev, just run:
```bash
npm install
```

---

### 2. `winCodeSign` macOS symlink error (elevated privileges)

**Symptom:** electron-builder fails during packaging with an error like:
```
EPERM: operation not permitted, symlink '...' → '...'
```
or the `7za -snl` extraction fails requiring admin rights.

**Cause:** electron-builder downloads `winCodeSign` (the signing toolkit) and extracts it using `7za -snl`, which attempts to create real Windows symlinks for macOS `.dylib` entries in the archive. Creating symlinks on Windows requires elevated privileges.

**Fix:** `scripts/build-win.mjs` contains the `ensureWinCodeSign()` function that pre-extracts `winCodeSign` **without** the `-snl` flag (so macOS symlinks are silently skipped). electron-builder then finds the cache directory already present and skips its own extraction entirely.

7-zip exits with code 2 for the two skipped darwin symlinks — this is expected and treated as a warning, not a failure.

The cache lives at:
```
%LOCALAPPDATA%\electron-builder\Cache\winCodeSign\winCodeSign-2.6.0\
```
Delete this folder to force a re-extraction on next build.

---

### 3. `MUI_FINISHPAGE_RUN already defined` NSIS error

**Symptom:** NSIS compilation fails during packaging:
```
!define: "MUI_FINISHPAGE_RUN" already defined!
```

**Cause:** `build/installer.nsh` was re-defining `MUI_FINISHPAGE_RUN`, which electron-builder's own `assistedInstaller.nsh` already defines (it uses it to launch the app on finish). Defining the same macro twice is a fatal NSIS error.

**Fix:** Remove the `!define MUI_FINISHPAGE_RUN` line from `build/installer.nsh`. electron-builder's built-in definition already handles launching the app after install — no override needed.

---

### 4. Auto-update returns HTTP 404 (`Cannot find channel "latest.yml"`)

**Symptom:** The running app logs:
```
Cannot find channel "latest.yml" update info: HttpError: 404
```

**Cause:** The backend server has no static file route for the `/updates/` directory. electron-updater requests `https://valor.dawn-star.co.uk/updates/latest.yml` but the server returns 404.

**Fix (server-side):** Add a static file route in the backend's `index.js`:
```javascript
app.use('/updates', express.static(path.join(__dirname, '../releases')))
```
Then copy `dist/latest.yml` and `dist/VALOR-Setup-<version>.exe` from the build output into the server's `releases/` folder after each release. See the [Release pipeline](#release-pipeline-publishing-updates) section above.
