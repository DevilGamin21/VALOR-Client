# Server Claude Prompt — VALOR Desktop Client Integration

You are working on the **VALOR backend** (`f:\VALOR\backend`). The VALOR Desktop Client (an Electron app) connects to this backend at `https://apiv.dawn-star.co.uk`. After auditing the desktop client's API calls against the backend source, one backend change is required and a few existing endpoint shapes need confirming.

## Required change — serve update files as static assets

The desktop client uses `electron-updater` to check for new versions. It makes a GET request to:

```
GET https://valor.dawn-star.co.uk/updates/latest.yml
```

The backend currently has **no `/updates/` route**, so this returns 404 and the auto-updater logs:

```
Cannot find channel "latest.yml" update info: HttpError: 404
```

### What to add in `backend/src/index.js`

```javascript
import path from 'path'
import { fileURLToPath } from 'url'

const __dirname = path.dirname(fileURLToPath(import.meta.url))

// Serve auto-update files for the desktop client
app.use('/updates', express.static(path.join(__dirname, '../../releases')))
```

Place this route **before** any catch-all / 404 handler. The `releases/` folder sits at the repo root (sibling to `backend/`). After each desktop release, copy two files into it:

| File | Generated by |
|------|-------------|
| `releases/latest.yml` | `electron-builder` (in client `dist/`) |
| `releases/VALOR-Setup-<version>.exe` | `electron-builder` (in client `dist/`) |

The desktop client's `electron-builder.yml` sets the update URL to `https://valor.dawn-star.co.uk/updates/`. The route above satisfies that.

> **No authentication needed** on `/updates/` — `electron-updater` makes unauthenticated GET requests to check for and download updates.

## Confirmed-working endpoints (no changes needed)

These were audited against the backend source. The desktop client calls them and they work correctly. Listed here so you know the desktop client depends on them and their response shapes must not change without updating the client.

### Auth

| Method | Path | Request body | Response body |
|--------|------|-------------|--------------|
| `POST` | `/login` | `{ username, password }` | `{ token: string, user: { id, username, email, isAdmin, avatarUrl, tier } }` |
| `POST` | `/logout` | — | `{ success: true }` |

The desktop client stores the JWT from `token` in `electron-store` (persists across restarts). The `Authorization: Bearer <token>` header is attached to all subsequent requests.

### User / profile

| Method | Path | Request body | Response body |
|--------|------|-------------|--------------|
| `GET` | `/me` | — | `{ success: true, user: { id, username, email, isAdmin, avatarUrl, tier, ... } }` |
| `PATCH` | `/profile` | `{ username?, email?, password?, currentPassword? }` | `{ success: true, user: { ... } }` |
| `POST` | `/profile/avatar` | `multipart/form-data` with field `avatar` | `{ success: true, user: { avatarUrl: '/api/avatars/<filename>', ... } }` |

**Avatar URL note:** The backend stores avatarUrl as `/api/avatars/<filename>` (the Next.js proxy path). The desktop client rewrites this to `https://apiv.dawn-star.co.uk/avatars/<filename>` because it calls the backend directly (not through the Next.js frontend). The backend already serves static avatar files at `GET /avatars/<filename>` — this is correct and must not change.

### Media

| Method | Path | Response body |
|--------|------|--------------|
| `GET` | `/movies` | Array of movie objects |
| `GET` | `/tv` | Array of TV show objects |
| `GET` | `/movies/:id` | Single movie object |
| `GET` | `/tv/:id` | Single TV show object |

### Watchlist

| Method | Path | Request body | Response body |
|--------|------|-------------|--------------|
| `GET` | `/watchlist` | — | `{ success: true, items: [ ...UnifiedMedia ] }` |
| `POST` | `/watchlist` | `{ mediaId, mediaType }` | `{ success: true, items: [ ...UnifiedMedia ] }` |
| `DELETE` | `/watchlist/:id` | — | `{ success: true, items: [ ...UnifiedMedia ] }` |

**Important:** All three watchlist endpoints return the full updated `items` array wrapped in `{ success, items }`. The desktop client relies on this shape.

### Admin

| Method | Path | Notes |
|--------|------|-------|
| `GET` | `/admin/users` | Requires `isAdmin: true` on the authenticated user |
| `PATCH` | `/admin/users/:id` | Update user fields |
| `DELETE` | `/admin/users/:id` | Delete user |
| `GET` | `/admin/media` | Media management |

## Field mapping — `isAdmin` vs `role`

The backend uses `isAdmin: boolean`. The desktop client's internal `User` type uses `role: 'admin' | 'user'`. The client normalizes this in `api.ts` — the backend does **not** need to change its field name.
